go源码阅读之WaitGroup

发表于 2020-01-08 分类于 go源码

  在 go 中,当我们需要等待一组 goroutine 完成再退出,可以使用 waitgroup 来进行阻塞等待进程的完成.通过在每个 goroutine 前将等待计数加一,并在 goroutine 执行完时将计数器减一,这样我们可以在需要等待的 goroutine 中使用 Wait() 等待所有的 goroutine 执行完毕后再继续执行或退出.
  这样做的主要原因时因为在 go 中,主协程退出后,所有子协程会立即终止,这样就需要 waitgroup 来保证子协程的运行.和主协程不同的是,在子协程中创建新的子子协程的话,当子协程退出后并不会立即终止其创建的子子协程,但在某些情况下,WaitGroup 还可以用来等待一系列操作的完成.

go源码阅读之timer

发表于 2020-01-05 分类于 go源码

  本次选择了 timer 进行解析，首先 go 的定时器是一个较为单一的模块，使用起来也十分的简单。其次，在当前最新的版本 1.14 中对 timer 进行了优化，修改了一些实现细节。

发展历程

  • 1.10 版本之前，定时器实现为一个全局的 timer 堆，所有的定时器进行统一的管理。这就导致了多线程环境下会造成频繁的加锁操作，导致性能较低
  • 1.14 版本之前，实现多个(写死为 64 个)全局的四叉堆，不同的 P 根据 id 的不同会使用不同的堆进行定时操作，虽然仍然需要对每个堆加锁，但是因为一个进程中 P 的数量一般不会大于 64，所以因为锁的粒度得到了降低，对性能影响不大，但是因为每个堆单独由一个协程进行触发操作，所以在上下文切换方面仍然有性能缺陷
  • 1.14 版本 timer 由 netpoll 进行驱动，将每个 timer 堆附着在 P 上，这就避免了在唤醒 timer 时导致的 M/P 切换的开销，并且因为引入了 netpoll 管理 timer，让定时器的触发更加准确
