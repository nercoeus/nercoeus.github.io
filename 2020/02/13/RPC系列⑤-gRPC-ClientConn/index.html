发表于 2020-02-13 更新于 2020-02-16 分类于 RPC系列

客户端的创建

  上一节讲了 gRPC 服务器的构建和对连接的处理。既然服务器已经准别就绪了，下面就来看看客户端是怎么去调用服务器的接口的。同样根据官方例子中的 client.go 可以看出使用客户端去调用服务器接口的步骤如下：

  1. 创建一个和服务器的连接 ClientConn
  2. 连接成功后，使用这个连接创建对应服务的客户端
  3. 使用客户端去调用服务器提供的接口即可

1 ClientConn 结构

  gRPC 使用 ClientConn 来管理连接，表示一个到服务器的逻辑连接，因为里面可能包含多条和服务器的连接通过负载均衡，用来执行 RPCs 方法。一个 ClientConn 可以根据 配置，负载 建立多条和服务器之间的连接。ClientConn 封装了一系列的功能，其中包括 名称解析、TCP 连接的建立和 TLS 握手。如果已建立连接出现了错误会重新解析名称并重新连接服务器。ClientConn 结构体如下：

这里比较重要的有：

  • conns ：用来管理一系列的 addrConn addrConn 其实就是一个和服务器的真正连接。
  • blockingpicker ：负载均衡管理器，通过 Pick() 来获取合适的连接执行调用。

  每个 ClientConn 对应多个 SubConn，然后基于 resolver 得到多个 SubConn。然后根据负载均衡获取使用的 SubConn。大致了解 ClientConn 后就看看客户端是怎么创建连接的。

1.3 getTransport

  后面会讲到在客户端调用 rpc 接口时会调动 NewStream() 创建一个和服务器之间的数据流对象。这个对象在创建时会调用 ClientConn.getTransport() 来通过负载均衡的方式获取一个 SubConn 对象用来真正的服务于这个 rpc 调用的数据传输。可以看到 getTransport 中是通过 Pick() 去真正的获取一条连接。

整个调用过程如下：

  1. rpc 调用接口 ->
  2. grpc.NewClientStream() ->
  3. ClientConn.NewStream() ->
  4. newClientStream() ->
  5. newAttemptLocked() ->
  6. getTransport() ->
  7. Pick()

  最后的 Pick() 根据注册的负载均衡器的不同实现也不相同，但结果就是返回一个 SubConn 用来作为 rpc 调用时的数据传输流。

3 调用服务端的 rpc 接口

gRPC 服务器提供的接口总的来说分为两种，这两种接口的调用方式是不同的：

  • 普通 rpc 接口，请求-应答 模式
  • 带数据流的 rpc 接口

3.1 调用普通 rpc 接口

  在生成的代码中，客户端通过 Invoke() 函数来发送 rpc 请求，并在收到应答后返回。

小结

  在 gRPC 中客户端通过使用 ClientConn 来管理和服务器之间的逻辑连接，每一个 ClientConn 包含至少一个 SubConn(物理连接)。数据流则使用 ClientStream 结构进行管理。底层使用的是 HTTP/2 协议进行数据的传输。
