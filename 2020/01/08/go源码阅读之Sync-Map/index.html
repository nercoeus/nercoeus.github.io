go源码阅读之Sync.Map

发表于 2020-01-08 更新于 2020-01-10 分类于 go源码

  gopher 估计对 sync.map 都不陌生，虽然其提供的接口有限，但作为 go 中为数不多的容器乃至唯一的可并发容器，了解一下 sync.map 的内部实现还是很有必要的，可以帮助你很好的定位到 sync.map 在实际场景中的不足之处.已经它的高效使用方式.

sync.map 源码分析

sync.map struct

  sync.map 使用了 read 和 dirty 两个 map 结构使整个 map 的操作尽量避免加锁,只读的或者删除的情况下都可能避免加锁,通过对 read 的原子操作即可满足一些操作.并通过 misses 来判断是否需要将 dirty 同步到 read.使用 amended 来表示 dirty 上有没有新键值对

Load 操作

Load 操作有几种情况:

  1. read 中不存在目标数据,且 dirty 中存在未同步数据
    • 从 dirty 中尝试获取 dirty[key],查找不到但会查找失败,否则返回目标指针
  2. read 中不存在目标数据,且 dirty 不存在未同步数据
    • 返回查找失败
  3. read 中存在目标数据
    • 从 read[key] 中加载数据,如果是删除状态就返回查找失败,否则返回目标指针

  这里需要注意,当 sync.map 的操作穿透了 read 需要加锁查找 dirty 时,都需要在加锁后重新获取 read 的情况,因为在加锁的过程中可能发生了 dirty 的数据提升至 read.如果不检查的话就可能错过数据.

Store 操作

Store 操作流程如下:

  1. read 中存在这个键值对(表示原来存储过),先使用 tryStore 尝试存入 read
    • read 中的旧键值对如果被删除可能有两种情况
      • p == nil 这表示仅仅在 read 中被删除了,这种情况直接复用 read 中的键值对即可返回操作成功
      • p == expunged 此时不能简单地修改 read 完成值的存储,需要在后面加 mu 锁并将 p 设置为 nil,并同时添加键值对到 dirty 中,这是因为 p == expunged 代表 dirty 中没有对应 key 的槽位,所以这个操作对 read 是复用,但对于 dirty 是添加
  2. 不存在于 read 中,但是 dirty 中存在,直接修改 dirty 中的值,这是因为这个键值对还没有同步给 read
  3. 或者两个 map 中均不存在,需要先设置 amended,并保证 dirty 不为空,然后把键值对存入 dirty

LoadOrStore 操作

看懂了上面的 Store 操作, LoadOrStore 也类似,只是多了一些判断,整个逻辑十分好理解

Delete 操作

通过 Delete 将键值对的值设置为 nil,先看 read 再看 dirty

Range 操作

  依次对 map 中的键值对执行 f(),返回 false Range 就会停止.Range 的时间复杂度可能会是 O(N) 即便是 f 返回 false.
  如果 dirty 和 read 中的数据不同步,会先从 dirty 中把数据复制一份到 read 中,然后再进行 Range

小结

  sync.map 使用两个 map 并通过一定的条件进行二者之间的同步,从而避免大多数情况的加锁操作,值得注意的是,dirty 拥有 read 中的所有键值对(除了被标记为 expunged 的键值对),并还可能存在最新的值.
  当 map 的操作透过 read 访问 dirty 的次数 >= len(dirty) 就会把 dirty 中的数据复制一份到 read 中,并在向 dirty 中添加数据时,把 read 中的键值对(除了标记为删除的)同步到 dirty 中
