go源码阅读之Pool

发表于 2020-01-08 更新于 2020-01-09 分类于 go源码

  sync.Pool 和 Mutex,Cond 等同步原语或锁不同,它是用于存储数据的结构体.在 go 中实现的 pool 可以用来存储临时的一类对象,如果有需要时会从 Pool 中获取一个对象进行使用,当不使用时会从新放回到 Pool 中,大大减少了每次减少时均进行对象的初始化过程浪费的性能.在 cpp 中也存在线程池这种东西,不过 go 因为有 goroutine 所以并不需要线程池,但还是可以作为连接池等等进行使用.

Pool 源码分析

Pool struct

  pool 是一组可以单独保存和检索的临时对象,Pool 的目的是缓存已分配但是没有使用的对象以供以后进行使用,从而减轻内存分配器以及 GC 的压力.同时,这也使得构建高效且线程安全的空闲列表变得容易,但它并不适用于所有空闲列表.
  它的正确用法是管理一组临时项,他们在程序中是并发独立的,客户端之间共享这些对象并且可以被重用,Pool 提供了一种将分配开销平摊到所有客户端的方式
  在 fmt 包中,很好的使用了 Pool,fmt 维护了动态大小的临时输出缓冲区存储.缓冲区会增大(过多 goroutine 进行打印操作时)以及空闲时缩小.
  另外,为短期对象维护 Pool 并不合适,因为这不能很好的分摊性能,应该让它们自己维护空闲链表更加高效
  首先 Pool 中其实为每个 P 维护了一个结构,所以系统中有多少个 P,就有对应个数的 poolLocalInternal

Put 操作

  先来看看 Put 操作,将对象放回到 Pool 中.优先存入 private,如果 private 已经存在值了,就要尝试放入 shared

Get 操作

用户使用 Get 来从 Pool 中获取一个对象,主要过程如下:

  1. 从 lcoalPool 的 private 获取,成功直接返回
  2. 从 localPool 的 shared 获取,成功则返回,失败进入 getSlow
  3. 从别的 P 的 localPool 的 shared 偷取末尾节点对象,成功则返回
  4. 从自己的 victim 中尝试获取 private 对象,成功则返回
  5. 把自己作为第一个节点的数组,开始从 victim 的 shared 中获取对象,成功则返回
  6. 那都拿不到,New 为空返回,不为空使用 New() 创建一个对象并返回

  这里不用担心从别的 P 的 localPool 偷取会造成同步问题,因为 shared 的实现是一个尾部并发安全的链式队列,interesting,不过暂时不放在这篇博客中进行讲解

Pin 操作

  使用 Pin 操作来将 goroutine 和 P 进行绑定,防治抢占,根本原因还是因为 Pool 中是对每个 P 单独维护的数据结构保存对象,go 的源码为了 P 真是做了很多的修改,哪里都可以看到不同 P 之间并发操作的设计和优化

与 GC 相关联

  这里通过在初始化的时候将清理函数注册到 GC 中来实现 victim 优化.这种用法可以带来的好处如下:

  • 降低了 GC 发生前清理当前周期中产生的大量缓存对象,某些情况下可以进行二次利用
  • 降低了 New 使用的频率

poolChain

  看一下 poolChain 的 history 可以知道这是 19 年初新添加专门给 Pool 使用的组件.作为 Pool 的底层结构，用来存储池中的对象，代码值得一看。

poolDequeue struct

  从下往上看,先看 poolChain 的基石 poolDequeue,poolDequeue 是一个无锁且固定大小的单生产者多消费者的循环队列.单个生产者可以从头部 push 和 pop,消费者可以从尾部 pop

poolDequeue 的操作

  因为将 head 和 tail 进行组合打包存储,所以提供了 unpack 和 pack 来进行操作.unpack 将 headTail 分开并返回,pack 则返回组合好的结果,操作就是简单的位运算,不细说

在 poolDequeue 用户分为生产者和消费者,总共提供三种操作如下:

  1. pushHead 仅供单生产者使用
  2. popHead 仅供单生产者使用
  3. popTail 供多消费者使用

poolChain struct

  整个 poolChain 就是一个基于 poolDequeue 的链表，操作虽然是对链表的操作，但操作的是底层的 poolDequeue

poolChain 的操作

  使用 poolChain 首先需要明确一个前提就是它是尾部并发安全但是头部并不是并发安全的,好在 Golang 中仅在 sync.Pool 中使用了 poolChain,且严格遵守了这两点
  go 中使用 poolChain 而不是直接使用 poolDequeue,是为了动态的增长链表的长度,因为 poolDequeue 底层使用的是数组,所以使用了相对内存友好的链表结构堆 poolDequeue 进行包装,将每个数组的长度限制为 8,可以很好地利用内存
  因为链表使用的是 poolDequeue,所以链表尾部仅仅使用了简单的 CAS 操作即可保证并发安全,因为 poolDequeue 已经保证了尾部删除的并发情况

  这里说的单生产者在 Pool 中对应的是拥有 LocalPool 的 P,多消费者就是其余的 P 可以从 Pool 中偷取对象.

小结

  Pool 通过为每个 P 维护一个对象队列,并且进行 Pin 操作,可以实现无锁的情况下获取自身的队列,并且,添加了 private 来进一步优化,保证一般情境下完全不需要加锁便可以快速获取一个对象.同时还实现了 victim 用来防止从 localPool 获取不到对象并且也偷不到对象,可以从上一个 GC 周期放入 victim 的缓存获取,完成了类似于二级缓存的操作.
