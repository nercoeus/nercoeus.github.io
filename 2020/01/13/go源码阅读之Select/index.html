go源码阅读之Select

发表于 2020-01-13 分类于 go源码

  上一篇刚看了 Channel 的实现,现在趁热打铁,把 Select 一起看了,在 go 中 select 的使用离不开 channel.channel 之于 select.就好像 c 中 fd 之于 select.功能也基本相似.在 go 中,select 关键词会在编译时期转换为 selectgo()函数的调用
  虽然 select 表面上和 switch 有着相同的使用结构,但是其中一个不同之处在于,switch 中是按照 case 顺序进行比较执行,而 select 中同一时间多个 channel 触发并不会按照顺序,而是会提前进行随机排序然后按照这个随机顺序依次进行判断是否触发.

select 源码位于 go/src/runtime/select.go 中,下面开始进行分析吧

Select 源码阅读

select struct

select 其实并不存在结构一说,但是需要注意的是 select 中的 case 确实有结构的,如下:

select 在编译阶段进行了优化,总共有四种情况:
1: 空结构体 select{} 会直接阻塞 goroutine : waitReasonSelectNoCases
2: 只包含一个 case,会将 select 转化为 if 语句
3: 包含两个 case,其中一个是 default,会转化成 if{}else{}结构
4: 只有正常的 select 逻辑,会将其中所有 case 转化为 scase 结构体,调用运行时函数 selectgo 选取触发的 scase 结构体.通过 for 循环生成一组 if 语句,来判断是否选中 case

selectgo 操作

了解了 select 的结构,下面直接看看 selectgo 的代码

先来看看 selectgo 的参数:

  • cas0 指向的是 [ncases]scase 类型的数组,里面是所有的 case 对应的 scase 类型(如果有的话)
  • order0 指向的是 [2*ncases]uint16 类型的数组,前一半是 poll order,后一半用来 lock order
  • ncases 就是 cas0 指向数组的长度

小结

  整个 selectgo 函数有 380 行,即使把后面的不同情况的处理部分,仅就前面 select 的处理也有 280 行,想要让整个流程了然于胸还是有些困难的,下面就对 select 进行一个流程总结:

  1. 首先现随机生成所有 case 的遍历顺序,以及加锁顺序,初始化完毕
  2. 遍历所有 channel,如果可以立即出发就根据对应的情况跳转到后面对应的执行情况中执行并返回即可
  3. 所有 case 都没有执行,且存在 default,默认返回并触发 default
  4. 将 goroutine 阻塞在所有的 channel 的(接收队列或发送队列)中,并陷入睡眠等待触发
  5. 唤醒后,继续加锁遍历所有 case,选择触发的 sudog 并返回对应的索引
