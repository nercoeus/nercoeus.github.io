go源码阅读之timer

发表于 2020-01-05 分类于 go源码

  本次选择了 timer 进行解析，首先 go 的定时器是一个较为单一的模块，使用起来也十分的简单。其次，在当前最新的版本 1.14 中对 timer 进行了优化，修改了一些实现细节。

发展历程

  • 1.10 版本之前，定时器实现为一个全局的 timer 堆，所有的定时器进行统一的管理。这就导致了多线程环境下会造成频繁的加锁操作，导致性能较低
  • 1.14 版本之前，实现多个(写死为 64 个)全局的四叉堆，不同的 P 根据 id 的不同会使用不同的堆进行定时操作，虽然仍然需要对每个堆加锁，但是因为一个进程中 P 的数量一般不会大于 64，所以因为锁的粒度得到了降低，对性能影响不大，但是因为每个堆单独由一个协程进行触发操作，所以在上下文切换方面仍然有性能缺陷
  • 1.14 版本 timer 由 netpoll 进行驱动，将每个 timer 堆附着在 P 上，这就避免了在唤醒 timer 时导致的 M/P 切换的开销，并且因为引入了 netpoll 管理 timer，让定时器的触发更加准确

1.13 版本

对最新版本的 timer 进行解析之前，先来回顾一下 1.13 版本的 timer：
  在旧版本的 timer 中，使用了一个全局的结构体 timers 来存储所有(总共 64 个) timer 堆，不同的 P 会根据 id 简单的取模运算使用唯一的 timer 堆。每个 timer 堆底层使用四叉堆进行实现，最快将要触发的节点放在堆的根部(四叉堆得具体实现方式放在最后解析，现在只需要知道它是一个最小堆即可)
  并在每个 timer 堆创建的时候启动一个协程(timerproc)：用来循环检测，每次检测会触发堆中时间到达的 timer。然后会 sleep 到下一个 timer 触发时间点。
  1.13 版本的 timer 实现具有一个问题，就是使用 Ticker 和 Sleep 时会导致大量的上下文切换会消耗大量的性能,这是因为此时定时器堆是由 timerproc 来进行触发，在定时器休眠时，运行 timeproc 的 G 与 M 的结合体会释放 P，从而阻塞在系统调用上，然后从休眠中醒来时会找到一个新的 P 用来触发定时器，这种频繁的释放和寻找 P，会浪费大量的性能。某些情况下甚至会出现上下文切换时间高于睡眠时间。

Timer 实现

  在 1.14 版本中，对 timer 进行了大修改。每个 P 自己维护一个定时堆，而不是将堆统一进行全局存放。并且给定时器新添加了状态标志。还引入了 netpoll 来实现 timer 的高效触发。

数据结构

先来看看相关的数据结构：

添加一个 timer

将一个新建的 timer 通过 addtimer 添加到当前的 P 的定时器堆中

  至此，一个新创建的定时器已经添加到对应的 P 的堆上了，在添加的同时也会执行 cleantimers 和 wakeNetPoller 操作

删除一个 timer

想要删除一个定时器首先需要把它的状态设置为 Delete。
下面几种情况才会真正进行删除

  1. 会在 adjusttimers 的时候进行统一删除
  2. 在插入 timer 时通过 cleantimers 删除没用的 timer
  3. 在运行 timer 时删除头部的状态为 delete 的节点

顺便提一下，对于 timerModifiedEarlier 状态的定时器会先将其删除再添加回定时器堆中

修改一个 timer

  netpoll 通过 modtimer() 来对一个 timer 进行修改，可以对已经删除的节点进行复用，但是不能对堆中准备删除的节点进行复用，根据修改后的时间设置 timer 的状态为 timerModifiedEarlier 或 timerModifiedLater 即可，注意最后把触发时间添加到 netpoll 中

重置一个 timer

重置一个 timer 会复用已经删除和待删除的 timer。

运行一个 timer

  由 checkTimers() 调用 runtimer 对定时器堆从头部开始遍历找到一个可触发的 timer，同时对等待修改和删除的 timer 进行调整和删除

触发 timer

checkTimers

  因为每个 G 在调度完成之后，会重新进入调度循环。此时会对 timers 进行检查，执行 checkTimers。
首先对 timers 进行调整，然后对可触发的 timer 进行执行

  除了通过调度器在调度的过程中循环触发 timer，在添加 timer 或者重置 timer 的过程中可以使用 wakeNetPoll 来唤醒调度循环。从而可以快速的进入 checkTimer() 来检查是否有 timer 需要触发，提高的 timer 的触发精度

小结

  新版本的 timer 基本解决了上一版的的性能问题，但是同时也变得更加复杂。通过每个 P 自己维护一个定时器堆，将旧版本的 timerproc 干掉。使用 P 的调度循环直接对 timer 进行触发，同时使用 netpoll 来触发 timer，这也让新版本的定时器更加的具有时效性以及更加节约性能。
